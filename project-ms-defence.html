The Incident

In April 2025, Marks & Spencer suffered one of the most damaging cyberattacks in UK retail history, not because of a sophisticated zero-day exploit, but because of a phone call. This project started as a question: what preventable failures led to £300m in lost profit, and what would a defence tool built on those failures actually look like?

The Attack Kill Chain

The breach unfolded across several months, with the most critical failure happening not in a data centre, but on a support call. Understanding exactly how the chain broke reveals where the defences needed to be.

1
Initial compromise, February 2025
Scattered Spider gained early access and exfiltrated the Windows domain controller's NTDS.dit file, the Active Directory database storing password hashes for every domain user. This happened two months before the ransomware triggered, entirely undetected.
Stealth
2
Offline hash cracking
With NTDS.dit in hand, attackers cracked the hashes offline, outside M&S's detection perimeter, yielding plaintext credentials for a range of domain accounts.
Offline
3
Social engineering the helpdesk, April 17
An attacker impersonated an M&S employee and called TCS (Tata Consultancy Services), M&S's outsourced IT helpdesk. The agent reset the target user's password, handing over the critical entry point. No robust identity verification stopped them.
Root Cause
4
Lateral movement & privilege escalation
Using the reset credentials, attackers moved laterally across connected systems, escalating privileges and mapping M&S's digital infrastructure before triggering the payload.
Lateral
5
DragonForce ransomware deployed, April 19
The group deployed DragonForce ransomware in a double-extortion attack, encrypting servers and simultaneously exfiltrating customer data including names, addresses, dates of birth, and order histories.
Impact

The most dangerous attack isn't the one that exploits your technology. It's the one that exploits your people.

Research Methodology

This project synthesised findings from multiple cybersecurity analysts, the UK Parliamentary Business and Trade Sub-Committee hearing at which M&S Chairman Archie Norman confirmed the social engineering root cause, and technical analysis of the Active Directory exploitation pathway.

The goal was not to audit M&S specifically, but to extract a generalised model of the vulnerabilities and translate those into a tool any organisation could run against itself.

✕
Helpdesk identity verification protocol FAIL · Root cause
What this checks Whether your IT helpdesk requires a second factor of identity verification before performing privileged actions like password resets. Verbal confirmation of a name or employee ID is not sufficient. How M&S was exploited Attackers called TCS helpdesk, posed as an M&S employee, and convinced the agent to reset a domain account password. No robust identity check stopped them. Remediation
  • Require manager co-approval or a hardware token confirmation before any password reset
  • Implement a callback verification policy to the user's registered number
  • Log all helpdesk-initiated resets with a mandatory hold period
✕
NTDS.dit exfiltration monitoring FAIL · Stealth phase
What this checks Whether alerting exists for access to or copying of NTDS.dit, the Active Directory database storing all password hashes. This file should never leave a domain controller. How M&S was exploited Attackers exfiltrated NTDS.dit as early as February 2025, two months before the ransomware triggered — undetected the entire time. Remediation
  • Alert on Volume Shadow Copy Service usage on domain controllers
  • Monitor for unusual outbound traffic from DCs
  • Deploy a SIEM rule: any read of ntds.dit outside scheduled backup windows = immediate P1
✕
Third-party vendor privileged access controls FAIL · Supply chain
What this checks Whether third-party vendors have the minimum required access, and whether their actions are audited in real-time. Outsourced helpdesks with domain-level permissions are a significant attack surface. How M&S was exploited TCS had sufficient access to reset domain account passwords without secondary authorisation. The weak link was entirely outside M&S's direct control. Remediation
  • Apply zero-trust principles: just-in-time, just-enough access for all vendors
  • Require vendor actions to be co-authorised by an internal employee
  • Audit all privileged vendor activity in a centralised SIEM
✕
Privileged account MFA enforcement FAIL · Escalation
What this checks Whether all domain admin and privileged accounts require phishing-resistant MFA, not SMS or TOTP, which can be bypassed through credential theft and social engineering. Remediation
  • Enforce FIDO2/WebAuthn hardware keys for all domain admin accounts
  • Block legacy authentication protocols (NTLM, pass-the-hash mitigations)
  • Use Microsoft's Protected Users security group for tier-0 accounts
!
Lateral movement detection (east-west) Warning
What this checks Whether your network detection tools flag unusual authentication patterns, one account logging into many systems in quick succession is a classic lateral movement signal. Remediation
  • Deploy a User and Entity Behaviour Analytics (UEBA) tool
  • Alert on accounts authenticating to more than N systems within a time window
  • Apply network micro-segmentation to limit blast radius
!
Ransomware incident response playbook Warning
What this checks Whether a tested, documented ransomware IR playbook exists, including offline backups, communication templates, and escalation chains. Tested means tabletop exercises, not just documentation. Context Insiders suggested M&S lacked a proper ransomware IR plan, forcing improvised responses that stretched recovery to 46 days. Even basic preparation dramatically compresses recovery time.
!
Vishing simulation training for helpdesk Warning
What this checks Whether helpdesk and support staff receive regular, scenario-based training that includes vishing (voice phishing) simulations, not just email phishing awareness. Remediation
  • Run quarterly vishing simulation exercises with helpdesk staff
  • Train staff to recognise urgency and authority pressure tactics
  • Create a no-blame reporting culture for suspected social engineering calls
✓
Cyber insurance coverage in place Pass
Context M&S held cyber insurance of up to £100m. While it didn't prevent the attack, it provides partial financial recovery. Verify your policy covers ransomware, data exfiltration, and business interruption as separate line items, not all policies do.
✓
Regulatory breach notification readiness Pass
Context M&S notified the UK NCSC promptly and engaged external forensic experts. Under UK GDPR, organisations must report certain breaches to the ICO within 72 hours. Ensure your legal obligation checklist is pre-prepared, during an incident is not the time to draft it.

Key Lesson

The M&S breach is a case study in how sophisticated technical preparation, months of patient reconnaissance and offline cracking — can be undone in a single phone call. The most effective security investments are often the unglamorous ones: verification protocols, training, and access controls that make the human layer as hard to exploit as the technical one.
